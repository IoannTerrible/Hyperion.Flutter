# Инструкция: API для приёма логов клиента (backend)

## Контекст

Flutter-клиент отправляет содержимое лог-файла приложения на сервер по нажатию кнопки «Upload logs» в профиле. Реализуй на бэкенде API-эндпоинт для приёма этих логов.

## Спецификация API

### Метод и путь

```
POST /api/logs
```

### Авторизация

- **Bearer token** в заголовке `Authorization: Bearer <token>`
- Тот же токен, что используется для `/api/devices`, `/api/sessions` и т.п.
- При отсутствии или невалидном токене — **401 Unauthorized**

### Заголовки запроса

- `Content-Type: application/json`
- `Accept: application/json`
- `Authorization: Bearer <token>`

### Тело запроса (JSON)

```json
{
  "content": "<полный текст лог-файла>"
}
```

- `content` — строка, содержимое лог-файла приложения (текст).
- Формат строк в логе: `YYYY-MM-DD HH:MM:SS [ClassName] message`.

### Пример успешного ответа

- **200 OK** или **201 Created**
- Тело: пустое или JSON `{}` (или `{ "received": true }`).

### Примеры ошибок

- **401 Unauthorized** — отсутствует или невалидный токен.
- **400 Bad Request** — если `content` отсутствует или пустой.
- **413 Payload Too Large** — если размер тела превышает допустимый (например, 1 MB).

### Пример запроса (curl)

```bash
curl -X POST "https://your-api.example.com/api/logs" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <jwt_token>" \
  -d '{"content":"2025-02-24 12:00:00 [DevicesService] getDevices: calling API\n2025-02-24 12:00:01 [ProfilePage] ..."}'
```

## Что делать с логами на сервере

На усмотрение разработчика. Варианты:

1. **Сохранять в файл** — например, `logs/{userId}_{timestamp}.txt`.
2. **Писать в БД** — таблица `client_logs` с полями: `user_id`, `content`, `created_at`.
3. **Отправлять в систему логирования** — Elasticsearch, Loki, CloudWatch и т.п.
4. **Комбинировать** — сохранять в БД и дублировать в файл/лог-систему.

Рекомендуется привязывать логи к `user_id` из токена, чтобы можно было искать по пользователю.

## Важно

- **Не логировать токены и пароли** — клиент уже следует правилам (см. logging-rules.mdc) и не отправляет чувствительные данные.
- **Ограничить размер** — разумный лимит 512 KB–1 MB на запрос.
- **Rate limiting** — ограничить частоту запросов на пользователя (например, не чаще 1 раза в минуту).
